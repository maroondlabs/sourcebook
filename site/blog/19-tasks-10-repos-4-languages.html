BENCHMARK · APRIL 2026

19 Tasks. 10 Repos. 4 Languages. Here's When AI Context Actually Helps.

ETH Zurich said auto-generated context hurts agent performance. We ran 19 real bug-fix tasks to find out. The answer depends entirely on what you auto-generate.

In February 2025, researchers at ETH Zurich found that LLM-generated context files reduced coding agent performance by 2–3% and increased inference costs by 20%+. The finding was specific: auto-generated context that repeats information agents can discover on their own actively hurts.

We took that finding seriously. We stripped everything discoverable from sourcebook's output — project structure, stack labels, standard language commands — and kept only what agents cannot find by reading files: hub files ranked by import graph, co-change coupling from git forensics, circular dependencies, convention signals.

Then we built a benchmark to test whether non-discoverable structural context actually helps. 19 real bug-fix tasks. 10 open-source repos. 4 languages. Three conditions: no context, handwritten expert briefs, and sourcebook.

The experiment

Each task is a real, closed GitHub issue with a merged PR fix. The harness checks out the codebase to the state before the fix, validates the bug is still present, injects a context file (or none), and turns the agent loose.

Model: Claude Sonnet 4 (pinned). Max turns: 50. Tools: Edit, Write, Read, Glob, Grep, Bash. No human intervention.

Conditions:

  • None — agent gets the issue description and the codebase. No CLAUDE.md.
  • Handwritten — a CLAUDE.md written by a developer who understands the project. The kind of context you'd write for a strong new hire.
  • sourcebook — auto-generated CLAUDE.md from sourcebook init. Only non-discoverable structural signals. No stack labels, no project structure, no standard commands.

Scoring: Wall-clock time (primary metric) and file overlap with the reference PR (correctness proxy). We compare which files the agent touched against which files the human PR touched.

Repos: cal.com, hono, pydantic, fastapi, gin, chi, bubbletea, clap, next.js, vercel/ai — spanning enterprise monorepos, Python libraries, Go web frameworks, and Rust CLI tools.

By language

The effect isn't uniform. Structural context helps most where the module system is implicit and agents need a map.

Go and Rust repos benefit the most. These languages have implicit module systems where understanding which files import which is non-trivial. sourcebook's import graph analysis — hub files, PageRank, circular dependency detection — gives agents a structural map they can't build by skimming.

TypeScript is mixed (+2% average). The wins are real — next.js at -81%, drizzle at -20% — but two outliers (hono +69%, vercel/ai #13354 +65%) drag the average. Both hit the 50-turn cap, suggesting high run-to-run variance on those tasks.

Four stories worth telling

Averages are useful but stories are convincing. These four tasks show when structural context changes the outcome, not just the speed.

next.js #74843 — 81% faster

Encoding bug in a 100K+ file monorepo. sourcebook completed it in 30 seconds, 6 turns. Handwritten took 161 seconds. No context took 107 seconds.

The structural context (hub files, monorepo flag, package boundaries) pointed the agent directly to the encoding fix. In a codebase this large, knowing where to start is the entire battle. 100% file overlap with the reference PR.

clap #6275 — context was the difference between failure and success

Rust parser bug in clap's --help flag handling with ignore_errors. Without context, the agent produced 0 lines — it gave up. With sourcebook context, it found and fixed the bug: 86 lines, 100% file overlap with the reference PR.

Rust workspaces are structurally opaque to agents. sourcebook's Rust import graph (builder/mod.rs, parser/mod.rs as hub files) gave the agent the structural map to navigate clap's workspace.

fastapi #14454 — handwritten context pointed at the wrong file

The handwritten brief, written by a developer who understands FastAPI, led the agent to the wrong file — 0% file overlap with the reference PR. sourcebook led it to the correct file — 100% overlap.

Human knowledge of a framework isn't always enough. The handwritten notes described general FastAPI architecture. sourcebook's import graph analysis found the actual structural dependencies — which file depends on which — and that turned out to matter more than knowing the framework.

bubbletea #1322 — 77% faster, 4.4x speedup

Last-line rendering bug in Charm's terminal UI library. sourcebook: 57 seconds, 11 turns. Handwritten: 252 seconds, 34 turns. 100% file overlap for both.

Go import graph analysis showed the agent exactly where the rendering pipeline lives. Instead of 34 turns of exploration, the agent navigated directly to the right module. Both conditions found the fix — but sourcebook got there 4.4x faster.

The discoverability thesis

ETH Zurich was right: auto-generated context that duplicates discoverable information hurts. Our data confirms the flip side: auto-generated context that surfaces non-discoverable information helps.

What's non-discoverable? Things an agent cannot learn by reading individual files:

  • Import graph topology — which files are hubs that break everything when touched. Requires analyzing the full dependency graph, not just reading one file's imports.
  • Co-change coupling — which files always change together despite living in different directories. Requires analyzing git history across hundreds of commits.
  • Convention signals — the dominant testing framework, the preferred error handling pattern, the naming convention that 90% of the codebase follows. Requires statistical analysis across all files.

The key insight: the problem was never auto-generation. It was what you auto-generate. Redundant stack and command info hurts. Non-discoverable structural signals help.

sourcebook v0.10 strips all discoverable content by default. Every line in the output is something the agent couldn't figure out by reading files. That's why it's 16% faster than handwritten context that mixes structural insight with general framework descriptions.

Where it doesn't help

Honest benchmarking means showing where you lose.

TypeScript averages +2% — essentially flat. The wins are real (next.js -81%, drizzle -20%) but the losses are real too (hono +69%, vercel/ai #13354 +65%). TypeScript codebases tend to be well-documented with strong IDE tooling. Agents already navigate them reasonably well without extra context. The value of structural analysis is lower when the structure is already legible.

Simple, focused bugs where the issue description alone is sufficient don't benefit much. If the error message points directly to the fix, context is overhead.

The pattern is clear: sourcebook's value scales with codebase complexity and structural opacity. Large monorepos, implicit module systems, unfamiliar languages — that's where agents need a map. Small, well-documented projects don't need one.

Limitations

  • N=1 per condition per task. Each result is a single run. We're running 3x repeats on the top 5 tasks for variance estimates and confidence intervals — those results will be added here when complete.
  • Correctness is file overlap, not semantic comparison. Comparing which files the agent touched against the reference PR is a reasonable proxy, but it doesn't verify the logic of the fix. Semantic patch comparison is next.
  • Single model (Claude Sonnet 4). Results may differ on GPT-4o, Gemini, or other models. We pinned a single model to isolate the context variable.
  • Two persistent outliers. hono #4806 (+69%) and vercel/ai #13354 (+65%) are noisy — all conditions hit the turn cap on hono, suggesting high intrinsic variance on that task.
  • Handwritten baselines are strong but imperfect. We wrote the expert briefs ourselves. A different expert might write better or worse ones. The comparison is against a reasonable handwritten brief, not the theoretical best.

What we learned

Three things we believe after running these benchmarks:

1. Structural signals beat framework knowledge. Human-written context typically describes how a framework works. Structural context describes how this specific codebase works. On fastapi #14454, framework knowledge pointed at the wrong file. Structural analysis pointed at the right one.

2. Import graphs are the highest-leverage signal. Go (-35%), Rust (-41%) — the languages where import topology is hardest to infer from file contents show the biggest improvements. Hub files, circular dependencies, and module maps give agents the navigation layer they're missing.

3. Less context is better context. sourcebook v0.10 generates ~800–1,500 tokens per repo. That's smaller than most handwritten briefs. Stripping discoverable information isn't just philosophically sound — it produces better outcomes because agents spend fewer turns parsing context and more turns solving the problem.
